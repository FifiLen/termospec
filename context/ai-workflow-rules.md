# AI Workflow Rules

## Important: Historical notes are not implementation requirements

`progress-tracker.md` is a historical changelog. Do not resurrect old layouts, removed components, old feature specs, old hero sections, old Nuxt UI page rhythms, blueprint meshes, hatching systems, ProductPageHero variants, or previous catalogue experiments from it.

For implementation, use only:

1. the current user request,
2. the current feature spec,
3. `ui-context.md` current visual rules,
4. the actual existing code.

When a current user request conflicts with historical notes, the current user request wins.

## Approach

Build the Termospec website incrementally using a context-driven workflow. Context files define what to build, how to build it, what the current visual language is, and what still needs verification.

Always implement against these specs:

- `project-overview.md` — product/site goals and scope
- `architecture-context.md` — stack, boundaries, routing, content model
- `ui-context.md` — visual language, tokens, typography, layout rules
- `code-standards.md` — implementation conventions
- `content-map.md` — content structure, page sections, navigation model
- `progress-tracker.md` — current state, next unit, open questions

Do not infer or invent behavior when the context files already define it.

## Scoping Rules

- Work on one feature unit or subsystem at a time.
- Prefer small, verifiable changes over large speculative rewrites.
- Do not mix unrelated concerns in one step.
- Do not rebuild an entire page when the task asks for a single component or section.
- Do not change the stack unless explicitly requested.

## When To Split Work

Split an implementation step if it combines:

- global styling and page-specific component work
- navigation structure and content extraction
- CMS/content schema changes and visual redesign
- multiple unrelated pages
- SEO metadata changes and unrelated UI changes
- behavior that is not clearly defined in the context files

If a change cannot be checked quickly in the browser or by reading the component output, the scope is too broad.

## Handling Missing Requirements

- Do not invent product behavior, offer names, technical claims, certificates, parameters, or contact details.
- If a requirement is ambiguous, update `progress-tracker.md` with an open question before continuing.
- If public data conflicts with client-provided information, prioritize client-provided information and mark the conflict in `progress-tracker.md`.
- If old-site content is incomplete, preserve what is known and label the rest as `TODO: verify`.

## Protected Foundation

Do not modify generated or third-party foundation files unless explicitly requested.

Protected examples:

- UI primitives generated by a CLI or copied from a design system
- framework internals
- package lockfile except when dependencies intentionally change
- global config files unless the current unit specifically requires it

Project-specific styling and layout belong in app-level components, page components, data files, or content files.

## Keeping Docs In Sync

Update the relevant context file whenever implementation changes:

- Architecture, routes, content source, or deployment decisions → `architecture-context.md`
- Visual tokens, typography, components, animation style → `ui-context.md`
- Copy/content structure, sections, menu groups → `content-map.md`
- Current phase, completed work, next up, open questions → `progress-tracker.md`
- Code conventions → `code-standards.md`

`progress-tracker.md` must reflect actual completed work, not intended work.

## Before Moving To The Next Unit

1. The current unit works within its defined scope.
2. No invariant from `architecture-context.md` was violated.
3. Styling follows `ui-context.md`.
4. Content and labels are represented in `content-map.md` when they become part of the site.
5. `progress-tracker.md` has been updated.
